# cub3D

miniLibX で作る最初のレイキャスター

概要: このプロジェクトは、世界的に有名なゲーム Wolfenstein 3D に着想を得ています。これは史上初の FPS と見なされています。この課題ではレイキャスティングを探求します。目標は、迷路の中を進みながら道を見つける、動的な視点表現を作ることです。

バージョン: 12.0

## Chapter I - はじめに

Wolfenstein 3D は、著名な John Carmack と John Romero が率いる Id Software によって開発され、1992 年に Apogee Software から公開された、ビデオゲーム史上初の真の「ファーストパーソン・シューター」です。

図 I.1: John Romero（左）と John Carmack（右）

Wolfenstein 3D は、Doom（Id Software, 1993）、Doom II（Id Software, 1994）、Duke Nukem 3D（3D Realm, 1996）、Quake（Id Software, 1996）といった、ゲーム史に残る作品群の祖先にあたります。

今度はあなたが、この歴史を追体験する番です。

Wolfenstein 3D の舞台はもともとナチス・ドイツであり、不快に感じる可能性があります。本課題でこのゲームの画像や歴史に触れるのは、技術的な理由とポップカルチャー / ギークカルチャー上の理由によるものです。この作品はその両面で傑作と見なされているためです。

## Chapter II - 目標

このプロジェクトの目標は、1 年目の他の課題と同様に、厳密さ、C 言語の利用、基本アルゴリズム、情報収集などにあります。

また、グラフィック系の課題として、cub3D はウィンドウ、色、イベント、図形塗りつぶしなどの分野でスキルを伸ばす機会になります。

要するに cub3D は、細かな理論をすべて理解していなくても、数学の実践的で遊び心のある応用を探求できる優れた実験場です。

インターネット上にある多くの資料を活用しながら、数学を道具として使い、洗練され効率的なアルゴリズムを作っていきます。

興味があれば、課題を始める前にオリジナル版を試せます。

http://users.atw.hu/wolf3d/

## Chapter III - 共通指示

- プロジェクトは C で記述しなければなりません。
- プロジェクトは Norm に従って書かなければなりません。ボーナス用のファイルや関数も Norm チェック対象に含まれ、Norm エラーがあると 0 点になります。
- 関数は未定義動作を除き、予期せず終了してはなりません。たとえば segmentation fault、bus error、double free などです。これが起きた場合、プロジェクトは正常に動作しないものと見なされ、評価時に 0 点になります。
- ヒープに確保したメモリは、必要に応じてすべて適切に解放しなければなりません。メモリリークは許容されません。
- 課題で要求される場合、`cc` を用い、`-Wall`、`-Wextra`、`-Werror` を付けてソースを必要な出力にコンパイルする Makefile を提出しなければなりません。また、Makefile は不要な再リンクをしてはいけません。
- Makefile には少なくとも `$(NAME)`、`all`、`clean`、`fclean`、`re` のルールが必要です。
- ボーナスを提出する場合、Makefile に `bonus` ルールを含めなければなりません。このルールでは、メインパートでは許可されていないヘッダ、ライブラリ、関数などを追加します。課題に特別な指定がない限り、ボーナスは `_bonus.{c/h}` ファイルに配置しなければなりません。必須パートとボーナスパートの評価は別々に行われます。
- libft の使用が許可されている場合、そのソースと対応する Makefile を `libft` フォルダにコピーしなければなりません。プロジェクトの Makefile はまずその Makefile を使ってライブラリをコンパイルし、その後にプロジェクト本体をコンパイルしなければなりません。
- 提出不要で採点対象でもありませんが、テストプログラムを作ることを推奨します。自分や他の学生の成果物を簡単に検証できるようになり、とくに defense の際に役立ちます。defense では、自分のテストや、評価相手のテストを自由に使えます。
- 指定された Git リポジトリに成果物を提出してください。採点対象になるのは Git リポジトリ内の内容だけです。Deepthought が採点を担当する場合、相互評価の後に行われます。Deepthought の採点中にどこかのセクションでエラーが起きた場合、その時点で評価は停止します。

## Chapter IV - AI に関する指示

### Context

学習の過程で、AI はさまざまな作業を支援できます。AI ツールの機能や、それが自分の作業をどう支えられるかを探ってください。ただし、常に慎重に使い、その結果を批判的に評価してください。コード、ドキュメント、アイデア、技術説明のいずれであっても、自分の質問が適切だったか、生成内容が正しいかを完全には保証できません。見落としや誤りを防ぐうえで、仲間は非常に重要な存在です。

### Main message

- AI は反復的で退屈な作業を減らすために使うこと。
- コーディング、非コーディングの両面で、将来のキャリアに役立つプロンプトスキルを身につけること。
- AI システムの仕組みを学び、よくあるリスク、バイアス、倫理的問題を予測し回避できるようになること。
- 仲間と協力しながら、技術的スキルと対人スキルの両方を伸ばし続けること。
- 自分が完全に理解し、責任を持てる AI 生成物だけを使うこと。

### Learner rules

- AI ツールを試し、その仕組みを理解する時間を取り、倫理的に使えるようにし、潜在的なバイアスを減らすこと。
- プロンプトを書く前に、まず自分の問題をよく考えること。そうすることで、より明確で詳細かつ適切な語彙を使ったプロンプトを書けます。
- AI が生成したものは、常に確認し、見直し、疑い、テストする習慣を身につけること。
- 自分だけの確認に頼らず、必ず仲間のレビューを求めること。

### Phase outcomes

- 汎用的なプロンプトスキルと、分野固有のプロンプトスキルの両方を身につける。
- AI ツールを効果的に使い、生産性を向上させる。
- 計算論的思考、問題解決力、適応力、協働力をさらに強化する。

### Comments and examples

- 試験や評価など、実際の理解を示さなければならない場面に今後も繰り返し直面します。備えを怠らず、技術面と対人面の両方を磨き続けてください。
- 自分の考えを説明し、仲間と議論すると、理解の穴が見つかることがよくあります。仲間との学習を優先してください。
- AI ツールはあなた固有の文脈を持たないことが多く、一般的な回答を返しがちです。同じ環境を共有している仲間の方が、より的確で文脈に合った洞察をくれることがあります。
- AI は「もっともらしい答え」を返しがちですが、仲間は別の視点や重要なニュアンスを与えてくれます。品質確認の要として仲間を活用してください。

良い実践:

> AI に「ソート関数はどうテストすればいい？」と聞く。いくつか案が返ってくる。実際に試し、その結果を仲間とレビューする。一緒に方針を洗練する。

悪い実践:

> AI に関数全体を書かせ、それをそのままプロジェクトに貼り付ける。相互評価の場で、その関数が何をしているのか、なぜそうなっているのか説明できない。信用を失い、プロジェクトも落とす。

良い実践:

> AI を使ってパーサ設計の補助をさせる。そのあと仲間とロジックを順に確認する。2 つのバグを見つけて、一緒により良く、よりきれいに、完全に理解できる形へ書き直す。

悪い実践:

> Copilot にプロジェクトの重要部分のコードを生成させる。コンパイルは通るが、pipe をどう扱っているのか説明できない。評価で正当化できず、プロジェクトに失敗する。

## Chapter V - 必須パート - cub3D

- Program Name: `cub3D`
- Files to Submit: すべてのファイル
- Makefile: `all`, `clean`, `fclean`, `re`, `bonus`
- Arguments: `*.cub` 形式のマップ
- External Functions:
  - `open`, `close`, `read`, `write`, `printf`, `malloc`, `free`, `perror`, `strerror`, `exit`, `gettimeofday`
  - 数学ライブラリのすべての関数（`-lm`, `man 3 math`）
  - MiniLibX ライブラリのすべての関数
- Libft authorized: Yes

説明: 迷路の内部を、一人称視点から見た「現実的な」3D グラフィック表現として作成しなければなりません。この表現は、前述したレイキャスティングの原理を使って実装します。

制約は以下の通りです。

- miniLibX を使わなければなりません。OS に用意されている版を使っても、ソースから使っても構いません。ソースから使う場合は、Common Instructions で libft に対して述べられているのと同じルールを適用する必要があります。
- ウィンドウ管理は常に滑らかでなければなりません。たとえば、別ウィンドウへの切り替え、最小化などに正しく対応する必要があります。
- 壁の向き（North, South, East, West）に応じて異なる壁テクスチャを表示しなければなりません。テクスチャ自体の選択は自由です。
- 床と天井には、それぞれ異なる色を設定できなければなりません。
- プログラムはウィンドウ内に画像を表示し、以下のルールを守らなければなりません。
  - キーボードの左矢印・右矢印で、迷路内の視点を左右に向けられること。
  - `W`, `A`, `S`, `D` キーで、視点を迷路内で移動できること。
  - `ESC` キーでウィンドウを閉じ、プログラムを正常終了できること。
  - ウィンドウ枠の赤いバツ印をクリックしたときにも、ウィンドウを閉じ、プログラムを正常終了できること。
  - miniLibX の画像機能の使用を強く推奨します。
- プログラムの第 1 引数には、`.cub` 拡張子を持つシーン記述ファイルを受け取らなければなりません。
  - マップは 6 種類の文字のみで構成されなければなりません。`0` は空きマス、`1` は壁、`N`、`S`、`E`、`W` はプレイヤーの開始位置と向きを表します。

これは単純な有効マップの例です。

```text
111111
100101
101001
1100N1
111111
```

  - マップは壁で閉じられていなければなりません。閉じていない場合、プログラムはエラーを返さなければなりません。
  - マップ本体以外の各要素は、1 行以上の空行で区切って構いません。
  - マップ本体は必ず最後になければなりませんが、それ以外の各要素はファイル中の任意の順序で記述できます。
  - マップ以外では、各要素の情報は 1 個以上のスペースで区切って構いません。
  - マップはファイルに書かれている見た目のままパースしなければなりません。スペースもマップの有効な一部です。扱い方は実装に委ねられますが、ルールを満たす限り、どのようなマップでもパースできなければなりません。
  - マップ以外の各要素は、1 文字または 2 文字からなる種別識別子で始まり、その後に厳密な順序で固有情報が続かなければなりません。
    - 北テクスチャ:

      ```text
      NO ./path_to_the_north_texture
      ```

      - 識別子: `NO`
      - 北側テクスチャへのパス

    - 南テクスチャ:

      ```text
      SO ./path_to_the_south_texture
      ```

      - 識別子: `SO`
      - 南側テクスチャへのパス

    - 西テクスチャ:

      ```text
      WE ./path_to_the_west_texture
      ```

      - 識別子: `WE`
      - 西側テクスチャへのパス

    - 東テクスチャ:

      ```text
      EA ./path_to_the_east_texture
      ```

      - 識別子: `EA`
      - 東側テクスチャへのパス

    - 床色:

      ```text
      F 220,100,0
      ```

      - 識別子: `F`
      - 範囲 `[0,255]` の `R,G,B` 色: `0, 255, 255`

    - 天井色:

      ```text
      C 225,30,0
      ```

      - 識別子: `C`
      - 範囲 `[0,255]` の `R,G,B` 色: `0, 255, 255`

  - 必須パートの最小構成 `.cub` シーン例:

```text

SO ./path_to_the_south_texture
WE ./path_to_the_west_texture
EA ./path_to_the_east_texture
```

  - ファイル内にどのような種類の設定ミスがあっても、プログラムは適切に終了し、`"Error\n"` に続けて任意の明示的なエラーメッセージを返さなければなりません。

## Chapter VI - README 要件

Git リポジトリのルートには `README.md` を置かなければなりません。その目的は、プロジェクトを知らない人でも、課題の内容、実行方法、関連情報の参照先を素早く理解できるようにすることです。対象には、仲間、スタッフ、採用担当者などが含まれます。

`README.md` には最低限、以下を含めなければなりません。

- 1 行目はイタリック体で、次の文でなければなりません。`This project has been created as part of the 42 curriculum by <login1>[, <login2>[, <login3>[...]]].`
- プロジェクトの目的と概要を明確に説明する `Description` セクション。
- コンパイル、インストール、実行に関する必要情報を含む `Instructions` セクション。
- この分野の代表的な参考資料（ドキュメント、記事、チュートリアルなど）に加えて、AI をどの作業・どの部分に使ったかの説明を含む `Resources` セクション。
- 必要に応じて追加セクションが求められる場合があります。たとえば、使用例、機能一覧、技術的選択などです。その場合は下で明示されます。

## Chapter VII - ボーナスパート

ボーナスは、必須パートが完全である場合にのみ評価されます。

ここでいう完全とは、単に完成しているというだけではなく、誤った使い方のような厄介なケースでも失敗しないことを意味します。採点時に必須パートが満点でなければ、ボーナスは完全に無視されます。

ボーナス一覧:

- 壁との衝突判定
- ミニマップシステム
- 開閉できるドア
- アニメーションスプライト
- マウスによる視点回転

より高度なゲームは後で作れます。ここに時間をかけすぎないでください。

ボーナス実装のために、他の関数を使ったり、マップに新しい記号を追加したりしても構いません。ただし、その利用理由を評価時に正当化できる必要があります。また、必要に応じてシーンファイル形式を変更しても構いません。賢く進めてください。

## Chapter IX - 提出と相互評価

通常どおり、課題を自分の Git リポジトリに提出してください。defense で評価対象となるのは、リポジトリ内にある作業内容だけです。ファイル名が正しいか、よく確認してください。

評価中には、プロジェクトに対する短い修正を求められることがあります。たとえば、ちょっとした挙動変更、数行のコードの記述または書き換え、簡単に追加できる機能などです。

このステップはすべての課題に適用されるとは限りませんが、評価ガイドラインに記載がある場合は、対応できるよう準備しておく必要があります。

このステップの目的は、プロジェクトのある特定部分について、あなたが本当に理解しているかを確認することです。

修正は、自分が選んだ任意の開発環境で行えます。たとえば普段のセットアップで構いません。また、評価ガイドラインで特別な時間指定がない限り、数分で対応できる程度の内容である想定です。

たとえば、関数やスクリプトへの小さな変更、表示の修正、新しい情報を保持するためのデータ構造の調整などを求められる場合があります。

詳細（範囲、対象など）は評価ガイドラインで指定され、同じ課題であっても評価ごとに異なる可能性があります。
